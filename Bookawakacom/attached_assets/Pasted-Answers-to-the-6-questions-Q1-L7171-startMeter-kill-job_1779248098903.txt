Q2 — L4242 vehiclestatus='Assigned' after accept: optimistic or wait?
✅ Optimistic, on tap. Use 'Picking', not 'Assigned'.

Server convention is vehiclestatus: 'Picking' for an accepted job in progress (server.js:7725, 9101). 'Assigned' isn't in the server's vocabulary — driver app writing 'Assigned' would diverge from what dispatch UI reads. Use 'Picking'.
Tap → flip local state to Picking immediately + POST command:'accept'. If server returns non-2xx (e.g. version_conflict, forbidden), roll back to previous vehiclestatus. The server's acceptBooking updates ZONE_DRIVERS to Picking too — both writes converge on the same value.
Q3 — L7642 jobpickup/jobdropoff mirrors: keep for hail, drop for dispatch?
✅ Keep for hail, drop for dispatch.

Dispatch path: server stamps jobpickup/jobdropoff/JobphoneNo/jobname into online/.../current itself (server.js:6282, 7381, hail-fanout at ~6008 §FIX-HAIL/2). Driver app mirror is redundant.
Hail path: driver supplies pickup/dropoff because the trip starts at the driver, not from a dispatcher record. Server also writes these on /api/job/create source:hail (§FIX-HAIL/2), but the driver-side write is what seeds the value during the create round-trip. Safe to keep — they'll match.
TL;DR: drop the dispatch-side mirror. Keep the hail-side mirror.
Q4 — Ratings (L7458, 7460, 7470): direct write or command:'update'?
✅ Keep as direct Firebase writes for now. Ratings aren't in /api/job/command's state machine — they're post-trip metadata, not lifecycle. Wrapping them would require adding rating to the updateBooking whitelist and the booking event stream, which is bigger than the 22c scope. Phase 2 candidate. Exception: L7460 writes rating into allbookings/{cid}/{bid} directly — that's fine because it's after the trip is in Completed terminal state; server has already moved on. No conflict.

Q5 — L7559 driver-created bookings (pendingjobs/ set from jobs.tsx): in scope?

Q6 — L6759, 6885 jobs/ remove on shift teardown: safe to delete?
